

## Диагноз

Полностью согласен — проблема во фронтенде. После последних бэкенд-фиксов (Puppeteer, Jina, BUG1-4) пайплайн стал работать дольше и иногда отдаёт `progress_pct` рывками или с задержкой между апдейтами. Фронт это не вытягивает по нескольким причинам:

### Что реально ломает UX

1. **Жёсткие таймауты в `SiteCheck.tsx` теперь стреляют по живому скану.**
   После моего же предыдущего фикса там стоит:
   - `HARD_TIMEOUT_MS = 120_000` (2 минуты общий потолок)
   - `STALE_PROGRESS_MS = 60_000` (1 минута без изменений → стоп)
   
   Puppeteer + networkidle2 + 1500ms hydration + waitForSelector 5s + второй fallback 12s + LLM-вызовы (BUG2 geoHint) — реальный пайплайн теперь спокойно идёт **2.5–4 минуты**. Фронт убивает поллинг ровно посередине, ставит `scanError` и юзер видит зависший прогресс-бар.

2. **При срабатывании таймаута мы НЕ проверяем финальный статус.**
   Сейчас логика: «прошло 120с → показать ошибку». А скан в этот момент уже мог завершиться `done`, просто фронт перестал спрашивать. Пользователь видит «зависло», хотя результат лежит готовый в БД.

3. **Шкала «живёт своей жизнью», потому что `ScanProgress` имеет внутреннюю анимацию**, которая тикает независимо от `realProgress` с бэка. Когда реальный прогресс приходит реже чем раз в 2 секунды (а Puppeteer-этап как раз такой), бар анимируется вперёд, потом дёргается назад на реальное значение → визуальное «дрожание».

4. **Нет восстановления `scanId` из URL/localStorage.**
   Если юзер обновил страницу или вернулся — активный `scanId` потерян, поллинг не возобновляется, хотя скан в БД уже `done`. Отсюда «проверка прошла, посмотреть нельзя».

5. **`pollStatus` создан через `useCallback` с зависимостями `[navigate, toast]`**, но вызывается напрямую из `handleSubmit` без обновления — это ОК, но при таймауте мы просто `return`, не сохраняя scanId никуда. Юзер не может даже руками открыть результат.

## Что делаем (только фронт, бэкенд не трогаем)

### 1. `src/pages/SiteCheck.tsx` — поднять таймауты и добавить graceful recovery

- `HARD_TIMEOUT_MS`: **300_000** (5 минут) — реальный потолок Puppeteer + LLM
- `STALE_PROGRESS_MS`: **120_000** (2 минуты без апдейта)
- Перед тем как сдаться по таймауту — **сделать ещё один финальный `getScanStatus`** и если статус `done` — нормально редиректнуть на `/result/:scanId`
- При старте сохранять `scanId` в `sessionStorage` (`owndev_active_scan`), при `done`/`error` — чистить
- На монтировании `SiteCheck`: если есть `owndev_active_scan` и нет `?url=` в query — попробовать дочекать его статус, и если `done` — сразу редирект, если `running` — возобновить поллинг с восстановленным UI

### 2. `src/components/site-check/ScanProgress.tsx` — синхронизировать шкалу с реальностью

- Когда `realProgress` приходит — **жёстко синхронизировать** внутреннее значение с ним (не позволять анимации обгонять)
- Между апдейтами разрешить плавное движение **максимум до `realProgress + 3%`**, не дальше
- Убрать «самопроизвольное» движение бара когда реальный прогресс не меняется > 30 сек — пусть стоит на месте, иначе создаётся ложное ощущение работы
- Поднять порог появления кнопки «Отменить» с 60с до 90с (чтобы не пугать на нормальном Puppeteer-скане)

### 3. `src/lib/api/scan.ts` — устойчивее обрабатывать сетевые ошибки поллинга

- При network error в `getScanStatus` не молчать, а вернуть специальный объект `{ status: 'unknown', progress_pct: lastKnown }`, чтобы поллер мог отличить «бэкенд недоступен» от «скан реально завис»
- Это нужно потому что после Puppeteer-фикса первый запрос на `/status` иногда отваливается по таймауту fetch (бэк занят), и сейчас фронт просто `setTimeout(poll, 3000)` без логирования — пользователь не понимает что происходит

### 4. Финальная проверка — добавить «спасательный» баннер

Если по таймауту пришли в состояние ошибки, но `scanId` есть — показать в `SiteCheck.tsx` отдельный блок «Проверка идёт дольше обычного. **[Открыть результат, когда будет готов]**» со ссылкой на `/tools/site-check/result/:scanId`. Юзер сможет вручную проверить через минуту-две.

- Header, мобильную навигацию

## Проверка

1. `/tools/site-check`, ввести `owndev.ru` или другой SPA — скан должен спокойно идти 2-3 минуты, бар не должен дёргаться, по завершении — автоматический редирект на результат.
2. Во время скана обновить страницу F5 — должен восстановиться прогресс или сразу открыться готовый результат.
3. Если бэк реально завис на 5+ минут — увидеть кнопку «Отменить» (после 90с) и финальный fallback-баннер «Открыть результат, когда будет готов» со ссылкой.
4. Шкала: визуальная анимация не должна уходить вперёд `realProgress + 3%`.
